Stated assumptions

Before reading the gaps, here's what this engine assumes

  1. A1. Reconciliation period = calendar month (March 2025, UTC).
  2. A2. A transaction "belongs" to March based on its platform timestamp, not settlement date.
  3. A3. A settlement "belongs" to March if settlement_date falls in March.
  4. A4. Exact amount match tolerance = $0.00. We surface even sub-cent differences.
  5. A5. Rounding gaps = |platform − bank| > 0 and < $1.00, consistent with truncation at 2dp.
  6. A6. Any transaction prefixed REF- or with amount < 0 is treated as a refund and requires a corresponding settlement acknowledgment from the bank.
  7. A7. "Duplicate" = the same transaction_id appears >1× in the settlements file, regardless of batch.

Production limitations

What this engine would get wrong in production

1. Timezone naïvety. Timestamps are treated as UTC. A payment recorded at 23:55 PST is 07:55 UTC the next day — this engine would misclassify it to the wrong month. Production needs explicit timezone normalization before any date comparison.

2. Partial settlement batches. Some banks split a single transaction across two settlement batches (e.g. partial captures). This engine does a 1:1 match on transaction_id and would flag all partial settlements as duplicates or missing.

3. FX transactions. Currency conversion rates fluctuate. A $100 USD transaction settled as £78.42 looks like a $21.58 gap when converted back at end-of-day rates. This engine has no FX logic and would generate false positives for any multi-currency data.
